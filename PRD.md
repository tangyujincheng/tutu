# 兔兔信件小屋 - 产品需求文档 (PRD)

## 1. 项目概述

### 1.1 项目名称
兔兔信件小屋 (Bunny Letter Game)

### 1.2 项目背景
为用户打造一个可以和AI兔兔写信互动的温馨小应用。每次给兔兔写信，兔兔都会用温暖可爱的语气回信；如果很久不写信，兔兔也会主动想念主人并主动来信，创造一份治愈的情感陪伴。

### 1.3 目标用户
单用户场景 - 仅为用户个人使用，**不需要多用户支持**，保持轻量化设计。

---

## 2. 核心需求

### 2.1 用户写信 + AI自动回信
**功能描述：**
- 用户可以在前端给小兔子写信
- 信件发送后，后端调用AI自动生成一封软萌可爱风格的回信
- 所有信件保存在系统中，用户可以随时查看历史通信

**验收标准：**
- ✅ 用户输入信件内容后点击发送，成功保存用户信件
- ⚠️ AI生成回信功能受模型配置问题影响，待修复
- ✅ 前端可以完整展示所有信件历史

### 2.2 未读消息提示
**功能描述：**
- 当有兔兔的未读回信时，页面显示小红点提示
- 用户打开信箱查看后，自动标记所有未读为已读，小红点消失

**验收标准：**
- ✅ 未读回信数量 > 0 时正确显示小红点
- ✅ 用户打开信箱后自动标记所有未读为已读
- ✅ 标记已读后小红点正确消失

### 2.3 超时主动发信
**功能描述：**
- 如果超过 **2小时** 用户没有给兔兔写信，且最后一封信不是兔兔发送的
- 兔兔主动给用户写信（AI生成想念内容）

**验收标准：**
- ✅ 系统每分钟定时检查距离上次用户发信的时间间隔
- ✅ 满足条件（>2小时，最后一封不是兔兔发信）才触发自动发信
- ⚠️ 逻辑正确，但因AI配置问题无法实际生成内容

### 2.4 AI配置安全存储与后台管理
**功能描述：**
- AI接口的API密钥等敏感信息必须安全存储，绝对不能暴露到前端
- 提供后台配置页面，方便修改AI配置和提示词

**验收标准：**
- ✅ API密钥加密存储在后端SQLite数据库
- ✅ 前端API无法获取任何敏感信息
- ✅ 提供独立的后台配置页面，可以修改：
  - API地址
  - API密钥
  - 模型ID
  - 系统提示词（控制AI回信风格）
- ✅ API密钥在后台也仅显示掩码（xxxx...xxxx），完整密钥不暴露

## 3. 交互流程

```
用户打开页面
  ├─► 检查是否有未读回信 → 有则显示小红点
  ├─► 用户点击「查看信箱」 → 打开信箱弹窗
  │   └─► 显示所有信件列表 → 点击信件查看详情 → 自动标记未读
  └─► 用户点击「给兔兔写信」 → 打开写信弹窗
      └─► 输入内容 → 发送 → 后端保存并请求AI回信 → 返回结果 → 更新信箱

定时后台检查（每分钟执行一次）：
  ├─► 判断 是否距离上次用户发信 > 2小时
  ├─► 并且 判断 最后一封信不是兔兔发的
  └─► 满足两个条件 → 触发兔兔主动发信 → AI生成内容保存 → 更新未读计数
```

## 4. 视觉设计要求

### 4.1 整体风格
- **温馨治愈，软萌可爱**
- 暖色调为主，营造兔兔小屋温暖氛围
- 交互流畅，动画自然不突兀

### 4.2 核心元素
- 主体：一只可爱的兔兔插图
- 主要操作按钮：查看信箱、给兔兔写信
- 弹窗组件：信箱列表弹窗、写信弹窗
- 未读提示：小红点徽章

## 5. 技术方案

### 5.1 部署环境
- 部署服务器：`14.103.60.103`
- 部署路径：`/var/www/bunny-letter-game/`
- 使用 PM2 守护 Node.js 进程
- Nginx 反向代理到 Node.js 服务

### 5.2 技术栈
| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 后端 | Node.js + Express | 轻量简洁，适合小型项目 |
| 数据库 | SQLite | 单文件轻量级数据库，无需额外服务，适合单用户场景 |
| ORM | Prisma | 类型安全，开发效率高 |
| AI | 火山方舟 ARK 大模型 API | 使用 Doubao 模型生成回信 |
| 前端 | 原生 JavaScript + CSS | 不需要框架，保持项目极简 |

### 5.3 安全约束
- API密钥必须加密存储
- 敏感信息绝对不能暴露到前端
- 加密密钥从环境变量读取，不硬编码
